# 코드 정리 구분

## 요약

> 코드 정리를 하지 않는다면 모를까, 코드 정리를 하면 어딘가에서는 처리해야 합니다. 그러면 어디서 해야 할까요?
> 요약하면, 코드 정리는 별도의 PR로 만들고, 가급적 PR당 몇 개의 코드 정리만 넣습니다.

### 코드 정리의 종류

책에 의하면, 코드 정리에는 다음과 같은 종류가 있어요.

- 다양한 변경 필요성을 구분하지 않은 상태에서의 변경 시도 (코드 정리의 구분이 없으며, 순서가 없음, 단일 PR)
- 동작 변경과 구조 변경 (코드 정리의 구분은 있지만, 순서가 없음, 단일 PR)
- 순서를 부여한 동작 변경과 구조 변경 (코드 정리의 구분도 있고, 순서가 있음, 단일 PR)
- 별도의 PR에 포함된 동작 변경과 구조 변경 (코드 정리의 구분도 있고, 순서가 있음, 여러 PR)

### 정답은 없지만, 짧은 PR을 권장

PR을 나눌지 또는 한 개의 PR로 처리할지는 장단점이 있는 선택이에요.  
너무 커도, 작아도 리뷰 요청에 대해 팀원들의 외면을 받기 쉬워요.

다만, PR이 작을 수록 검토하는 시간은 단축된다는 점, 변화에 대한 목적이 뚜렷하기 때문에 장려해요.

## 참고자료

코드 정리에 관하여 참고하면 좋은 글을 몇 가지 적어보았어요.

### 1. 작게 쪼개기

[링크](https://soojin.ro/review/small-cls)

CL(Change List)를 작게 쪼개는 것이 왜 필요한 지에 대해 서술했어요.

별개로, 해당 블로그의 콘텐츠의 깊이가 있다고 느껴졌어요.

### 2. 공통시스템개발팀 코드 리뷰 문화 개선 이야기

[링크](https://techblog.woowahan.com/7152)

위의 글에서 영감을 받은 분께서 코드 리뷰를 팀에 적용하는 과정을 남긴 글이에요.

PR 리뷰가 늦는 원인을 고민하고, 이에 관해 개선한 내용을 기술했어요.

### 3. 효과적인 코드 리뷰를 위해서

[링크](https://engineering.linecorp.com/ko/blog/effective-codereview#keep-changes-small)

여러 근거를 토대로 코드 리뷰에 있어 단위가 작은 것이 좋은 이유를 서술했어요.

### 4. 코드 리뷰 in 뱅크샐러드 개발 문화

[링크](https://blog.banksalad.com/tech/banksalad-code-review-culture/)

뱅크샐러드 2.0을 도입하면서, 마주했던 형상관리의 어려움을 작은 코드 정리로 해결한 경험을 소개한 글이에요.

## 발제

### 코드 정리 구분 외에도, 우리의 코드를 더욱 잘 관리할 수 있는 방법은 어떤 것이 있을까요?

여러 참고자료를 보건대, 결국 코드 정리 역시 각자의 환경과 경험을 기반으로 이루어지는 것 같아요.

다만, 우리가 각자의 간접경험에서 우러나온 팁들을 모아놓는다면, 여러 환경에서도 대응 가능한 원칙을 수립할 수 있을 것 같아요.

각자가 경험상 추천하는 코드 관리 방법은 어떤 것들이 있나요?

# CHAPTER 18. 코드 정리의 일괄 처리량

- 일괄 처리당 코드 정리수가 많다면 충돌, 상호작용, 추측으로 인한 비용이 증가하고 이와 반대로 일괄 처리 규모를 축소한다면 검토 비용이 증가한다고 생각한다

- 즉, 검토하는데 있어 드는 고정 비용은 상당히 많기 때문에 일괄 처리당 코드 정리수를 늘리면서 이에 대한 타협접을 찾으려고 한다

- 히지만 실제로 일괄 처리당 코드 정리 수를 줄이면 검토 비용은 증가하는 것이 아니라 감소하게 된다

> 코드 정리 비용을 줄이고자 한다면, 코드 정리 개수를 늘려서 동작 변경에 소용되는 비용을 줄이세요. 그러면, 검토 비용을 줄일 수 있습니다

### 질문

> 팀에 신뢰와 강력한 문화가 있다면 코드 정리 후에는 굳이 검토할 필요가 없습니다. 검토하지 않더라도 코드 정리가 소프트웨어 안정을 해치지 않으면 상호작용의 위험이 줄어듭니다.

- 여기서 신뢰와 강력한 문화란 무엇일까요?

- "검토하지 않는다"라는 것이 "코드 리뷰를 하지 않아도 된다"와 동일한 의미는 아닐 것 같은데요, Github Action과 같은 도구를 사용해서 테스트 코드에 대한 CI를 작성하는 것도 강력한 문화를 구축하는 하나의 예시가 될 수 있을까요 ?

- 예를들어 "코드 정리를 한 PR은 기능 자체를 수정하지 않았기 때문에 테스트 코드 CI를 통과하면 머지를 해도 된다"는 식으로 규칙을 정할 수 있을까요?

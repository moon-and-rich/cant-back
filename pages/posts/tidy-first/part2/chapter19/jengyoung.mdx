# 리듬

> "코드 정리를 관리하는 기술 중에는 정리의 리듬을 관리하는 일도 있습니다."

## 설명

여기에서 정리의 리듬이란, 개발자가 작업량 대비 코드 정리의 설계를 가져가는 템포를 의미해요.

즉, **정리의 리듬이 개발자에게 부담이 가지 않는 선에서 흐름이 유지되어야 한다**는 것인데요. 저자는 한 번의 코드 정리에 1시간 정도를 권장해요.

그 이유는 무엇일까요? 바로, 소프트웨어 설계가 **길을 닦아놓는 과정에 가깝기 때문**이에요.

우리가 하루 내내 설계를 고민하고 짰어도, 결국 조직의 편의에 맞지 않는다면 그 설계는 의미가 없어지고 바뀌어요.

따라서 일단 조그마한 코드 정리들을 선행하면, 복잡했던 도로가 포장되고, 따라서 동작 변경에 대한 부담이 덜어지므로 리듬을 살려 개발할 수 있게 되는 것이죠.

## [예시] 코드 정리의 리듬을 살리기 위한 TODO와 FIXME

동작 변경에 앞서, 베이스가 되는 코드들의 정리를 모두 다 하면 좋겠지만, 우리는 시간의 제약을 받고 있어요.

따라서, 이럴 때에는 다음 작업자에게 어떤 구조 변경이 이루어져야 하는지, 어떤 동작 변경이 추후 개선되어야 하는지를 전달하는 주석이 좋은 것 같아요.

```ts
// FIXME: 한 페이지 내에, 여러개 useModalRegister을 사용하면 open 함수 호출시 가장 첫번째 등록한 key만 등록되는 문제
// TODO: context api 아닌, recoil로 전역상태로 관리하도록 한다
// NAME 황재영
const register: TModalContextState["register"] = (key, { visible = false }) => {
  setModals((state) => {
    return new Map(
      state.set(key, {
        visible,
      })
    );
  });
};
```

## 사유

사실 이 글에서 나오는 예시가 다소 번역체로 인해 이해하기 어려웠어요 🥲

다만 핵심은, 다음 문구인 것 같아요.

> "소프트웨어 설계는 프랙탈이므로 설계도, 크게도 작게 할 수 있습니다."

우리는 제한적인 리소스 안에서 최대한의 개발을 하려면, 당연히 작아도 구현이 되는 설계를 지향해야 해요.

따라서, 일단 각자의 컨텍스트에 맞춰 설계의 리듬을 길게, 짧게 유지시켜야 한다는 게 아닐까 생각하고 있어요.

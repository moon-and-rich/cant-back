# 얽힘 풀기

1. 어떤 코드의 동작 변경을 하고 있는 중
2. 변경을 쉽게할 코드 정리를 알고 있음
3. 코드 정리를 진행하고 테스트 케이스를 작성
4. 이제 동작을 변경해야 한다
5. 어랏 다시 코드 정리 할 곳이 생겼다

이렇게 한 시간을 씨름한 결과

- 변경 대상인 동작을 모두 알게 되었고
- 그 동작들을 쉽게 변경하려면, 어떤 코드를 정리해야 하는지 모두 알게되었으나
- 문제는 정리한 코드와 변경할 동작이 함께 얽혀 버렸다

그렇다면 세 가지 선택자기 있지만, 어느 하나도 매력적이지 않다

- 그대로 배포한다
- 코드 정리와 변경사항을 별도의 하나 이상의 PR로 나누거나 하나의 PR을 여러 번의 커밋으로 나눌 수 있다
- 진행 중인 작업을 버리고, 코드 정리를 선행하는 순서로 다시 시작한다

우리가 작성하는 코드는 컴퓨터에 지시할 뿐 아니라, 컴퓨터에 지시하려는 의도를 동료들에게 설명해야 한다. 그렇다면 "진행 중인 작업을 버리고, 코드 정리를 선행하는 순서로 다시 시작한다"을 선택하는 것이 썩 나쁜 선택지는 아니다.

> 실타래를 풀려면 실이 엉켜있다는 사실을 알아차려야 시작할 수 있습니다. 따라서 실타래를 풀어야 할 필요성을 더 일찍 깨달을수록 작업량은 적어질 것입니다

기존의 동작 변경을 하던 코드가 정리 우선되었을 때 깔끔하게 동작 변경이 가능하다면, 원래 시점으로 돌려서 정리 후 기능변경을 해보자는 뉘앙스로 들렸습니다

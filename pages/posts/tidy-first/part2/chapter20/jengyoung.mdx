# 얽힘 풀기

> 실타래를 풀려면 실이 엉켜 있다는 사실을 알아차려야 시작할 수 있습니다.
> 따라서 실타래를 풀어야 할 필요성을 더 일찍 깨달을 수록 작업량은 적어질 것입니다.

## 설명

우리는 왜 큰 단위의 PR을 하게 될까요?

이에 관련하여, 혹시 이러한 경험을 한 적이 있지 않았나요?

1. 변경 대상인 동작을 모두 알게 됐어요!
2. 동작 변경을 위해 어떤 코드를 정리해야 하는지 모두 알게 됐어요!
3. 그런데... 정리한 코드와 변경할 동작이 얽혀 버렸어요 🥹

이럴 때, 저자는 3번을 권장해요. 이유는 다음과 같죠.

- 다른 사람에게 이를 설명하는 것 또한 비용이에요.
- 다시 구현하면서 새로운 것을 발견할 가능성이 높아져요.
- 이를 통해 동일한 동작 변경을 하면서도 더 많은 가치를 끌어낼 수 있게 돼요.
- 차후 코드 설계에 관한 전략적 선택을 할 때에도 부담감이 덜해요.

따라서, 코드 정리와 동작을 빠르게 분리시킴으로써, 코드 정리와 동작 변경 간의 선후 문제를 풀어내어 비용을 절감할 수 있게 돼요.

## 사유

이해는 되는데, 다소 조직적인 관점에서 풀어내기 어려운 주제라 생각해요.

간혹, PR이 너무 무겁다고 느껴지는 건이 있다면, 과연 나는 3번의 방법을 팀원에게 요청할 수 있을까 싶기도 하구요.

이전의 챕터에서는 코드 정리에 1시간 이상을 사용하지 않는다고 했지만, 사실 이 방법은 자칫 1시간 이상의 코드 정리를 허용하는 것 같아요.

그래서, 처음에 작업자의 전반적인 작업 순서 설계를 미리 듣고, 사전에 얽힘 문제를 해결하는 게 가장 중요할 것 같다는 생각도 들었던 파트네요.

# 코드 정리 시점

## 요약

- **코드 정리를 하지 않아도 되는 경우**
  - 앞으로 다시는 코드를 변경하지 않을 때
  - 개선하더라도 배울 것이 없을 때
- **나중으로 정리를 미뤄도 되는 경우**
  - 정리할 코드 분량이 많은데 보상이 바로 보이지 않을 때
  - 코드 정리에 대한 보상이 잠재적일 때
  - 작은 묶음으로 여러 번에 나눠서 코드 정리가 가능할 때
- **동작 변경 후에 정리해야 하는 경우**
  - 다음 코드 정리까지 기다릴 수록 비용이 늘어날 때
  - 코드 정리를 하지 않으면 일을 끝냈다는 느낌이 들지 않을 때
- **코드 정리 후에 동작을 변경해야 하는 경우**
  - 코드 정리를 했을 때 즉각적인 개선 효과를 얻을 때
  - 어떤 코드를 어떻게 정리해야 하는지 알고 있을 때

## 설명

### 아예 안 한다면

앞으로 수정할 일이 없는 코드일 경우, 변경하는 비용 대비 얻을 수 있는 가치가 한정적이에요.

### 나중에 정리하기

> 먼저, 우리의 시간은 한정적이라는 사실을 인지해야 해요.

그리고, 우리가 만약 여유가 생긴다면 어떻게 개발하고 싶은지를 떠올려보는 거죠. 만약 현재와 상상이 다르다면, 우리는 바쁜 상황인 거죠.

한편, 여유가 되면 하고 싶은 개발 리스트가 곧 해당 케이스에 해당 돼요.

당장 필요하지는 않지만, 해놓고 나면 또다른 동작 개선에 도움을 주는 코드들처럼 말이죠.

### 동작 변경 후에 코드 정리

대개 급하게 동작 변경을 해야 하는 경우가 있어요.

이럴 경우에는 코드 정리를 해보았자, 다시 또 동작이 바뀔 가능성이 높아요.

그렇기 때문에 코드 정리가 간단하지 않다면, 동작 변경을 우선적으로 하는 것이 좋아요.

### 코드 정리 후에 동작 변경

간혹 코드가 복잡하다보면 정리의 흐름이 중요해질 때가 오는데요.

이럴 때, 동작 변경을 먼저 하게 되면 코드가 이해가 되지 않게 되거나, 코드 정리의 흐름이 끊겨서 리듬이 깨지는 경우가 발생해요.

만약 리듬이 깨졌을 때 일이 원활하지 않을 것으로 예상이 된다면, 코드 정리를 먼저하는 것이 좋아요.

## 사유

일부는 공감이 됐지만, 일부는 공감이 되지 않았어요.

예컨대, `코드 정리를 하지 않아도 되는 경우`를 볼까요?

일단, 다시는 바뀔 일이 없다면 코드 정리를 하지 않는 게 좋을 수 있어요. 이것은 이해가 돼요.

하지만, `배울 것이 없을 때`라는 말은 도저히 받아들이기 힘들어요.

이전 챕터에서는 코드 개선은 추후 `나아갈 길을 미리 닦아놓는 과정`이라고 조직적인 차원에서 이야기를 했었는데, 이 말은 너무 지극히 개인적인 차원에서 언급이 된 것 같아요.

기타 `코드 정리 후에 동작을 변경해야 하는 경우` 중, `어떤 코드를 어떻게 정리해야 하는지 알고 있을 때`라는 조건 역시 동작 변경의 긴급도에 따라 완벽히 대응되는 것은 아닌 것 같다는 생각이 들어, 조금의 디벨롭이 필요하다는 생각이 들었어요.

## 발제

위의 사유처럼, 코드 정리 시점에 관련하여 생각보다 더 많은 조건들이 존재할 것 같아요.
